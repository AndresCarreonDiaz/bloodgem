# Research: nano-banana-2-replicate

## Summary

The exact Replicate model is `google/nano-banana-2`, titled "Nano Banana 2 (Gemini 3.1 Flash Image)" — an official Google-published model (announced Feb 26, 2026) described on Replicate as "Google's fast image generation model with conversational editing, multi-image fusion, and character consistency." I pulled the live OpenAPI schema directly from Replicate's version endpoint (latest version id 402c3da8f794..., created 2026-06-19). Inputs: `prompt` (string, required), `image_input` (array of image URIs, up to 14, default []), `aspect_ratio` (enum of 15 values incl. `match_input_image` default, `1:1`, `2:3`, `9:16`, `16:9`, `21:9`, plus extreme `1:4`/`4:1`/`1:8`/`8:1`), `resolution` (enum `1K`/`2K`/`4K`, default `1K` — Replicate does NOT expose Gemini's 0.5K tier), `google_search` and `image_search` (booleans, default false), and `output_format` (enum `jpg`/`png`, default `jpg`). Output is a single URI string (one image per prediction), delivered via replicate.delivery URLs that expire after 1 hour. Replicate pricing is per output image by resolution: $0.067 (1K, "around 14 images for $1"), $0.101 (2K), $0.151 (4K). Account-level rate limits: 600 prediction-creates/min, 3000 req/min on other endpoints, HTTP 429 with `{"detail":"Request was throttled..."}` on breach.

API usage: POST https://api.replicate.com/v1/models/google/nano-banana-2/predictions with `Authorization: Bearer $REPLICATE_API_TOKEN`, `Content-Type: application/json`, and `Prefer: wait` (blocks up to 60s; if exceeded the prediction returns in "starting" state and must be polled via predictions.get). Body: {"input": {"prompt": "...", "image_input": ["https://..."], "aspect_ratio": "1:1", "resolution": "1K", "output_format": "png"}}. Node.js: `const output = await replicate.run("google/nano-banana-2", { input })` — because the output schema is a single URI (not an array), you get one FileOutput; write it with `await writeFile("out.png", output)` or read `output.url()`.

Transparency: NOT supported natively. Multiple independent sources confirm "Nano Banana doesn't output alpha channels. You get a flat RGB image regardless of what you ask for" — asking for `png` output does not add alpha. Two proven workarounds: (1) chroma-keying — prompt for "EXACT hex #00FF00 (RGB 0, 255, 0) with NO gradients, NO noise, NO texture" background plus "a clean white outline/border 2-3 pixels wide around every sprite", then HSV-key (hue ±22° of 120°, sat > 0.3, val > 0.3); (2) triangulation matting — generate on white, edit-pass to "Convert the white background to pure black. Preserve all foreground elements.", then compute `alpha = 1 - (white_pixel - black_pixel)/255`. Matting yields true soft alpha but costs 2x per asset and is capped at 2K (4K times out); community wrapper models `nano-banana-2-transparent` and `nano-banana-2-bg-remove` automate it.

Character consistency is Nano Banana 2's headline strength and is well-suited to sprite pipelines: reviewers found a single reference image held recognizable identity across ~10 repeated generations, and it maintains "visual consistency for up to 5 characters and detail fidelity for up to 14 objects" in one workflow — the verdict from head-to-head tests vs Nano Banana Pro is "reach for Nano Banana 2 when... you're working with character consistency across multiple generations." Practitioner sprite-sheet guides converge on: one character per sheet, 3-6 sprites per row and 2-3 rows max, per-frame explicit descriptions, "CRITICAL" + capitals for non-negotiables, a 3-view turnaround (front / 45° / 90° side) as the canonical reference fed back via `image_input`, frame-by-frame decomposition for complex actions (combo attacks drift badly if requested as one sequence), and content-aware slicing because "Nano Banana does not space sprites in a perfectly uniform grid, even when you ask for one."

## Key findings

- Exact model slug: google/nano-banana-2 (official Google model on Replicate, page title 'Nano Banana 2 (Gemini 3.1 Flash Image)'); latest version id 402c3da8f79427a67715983c7e97ba0a1f909fd0ae445b1041357c7718016946 created 2026-06-19; Google announced the model 2026-02-26 as its 'best image generation and editing model'.
- Full input schema (from Replicate's dereferenced OpenAPI schema): prompt — string, REQUIRED, 'A text description of the image you want to generate'; image_input — array of {type: string, format: uri}, default [], 'Input images to transform or use as reference (supports up to 14 images)'; aspect_ratio — enum ['match_input_image','1:1','1:4','1:8','2:3','3:2','3:4','4:1','4:3','4:5','5:4','8:1','9:16','16:9','21:9'], default 'match_input_image'; resolution — enum ['1K','2K','4K'], default '1K' (note: the Gemini API's 0.5K/512px tier is NOT exposed on Replicate); google_search — boolean, default false (web-search grounding); image_search — boolean, default false (Google Image Search grounding; auto-enables web search); output_format — enum ['jpg','png'], default 'jpg'.
- Output schema: {"type": "string", "title": "Output", "format": "uri"} — a SINGLE image URL per prediction (e.g. https://replicate.delivery/xezq/.../tmp5hj7uutl.jpeg). Replicate output file URLs expire after 1 hour, so download immediately.
- Replicate pricing (from model page pricing JSON): $0.067 per output image at 1K ('or around 14 images for $1'), $0.101 per output image at 2K, $0.151 per output image at 4K; metric is image_output_count, criteria keyed on 'target resolution'. Google-direct Gemini API is $0.045 (0.5K) to $0.151 (4K), with Batch API at half price ($0.022/$0.034/$0.050/$0.076), billed as image output tokens at $60.00 per 1M output tokens.
- Rate limits (Replicate docs, account-wide not per-model): create predictions at 600 requests/minute; all other endpoints 3000 requests/minute; short bursts above limits tolerated before throttling; 429 response body: {"detail":"Request was throttled. Your rate limit resets in ~30s."}; stronger limits apply as credit approaches zero; granted-credit accounts without a payment method are capped at 1 req/sec, max 6 req/min.
- HTTP API call: POST https://api.replicate.com/v1/models/google/nano-banana-2/predictions with headers 'Authorization: Bearer $REPLICATE_API_TOKEN', 'Content-Type: application/json', 'Prefer: wait' — Replicate docs: 'The request will wait up to 60 seconds for the model to run. If this time is exceeded the prediction will be returned in a starting state and need to be retrieved using the predictions.get endpoint.' Example body: {"input": {"prompt": "...", "image_input": ["https://..."], "aspect_ratio": "1:1", "resolution": "1K", "output_format": "png"}}.
- Node.js pattern: import Replicate from 'replicate'; const replicate = new Replicate(); const output = await replicate.run('google/nano-banana-2', { input: { prompt, image_input: [refUrl], aspect_ratio: '1:1', resolution: '1K', output_format: 'png' } }); then await writeFile('./output.png', output) — output is a single FileOutput (not an array) because the schema output is one URI string.
- Transparency: NO native alpha channel. 'Nano Banana doesn't output alpha channels. You get a flat RGB image regardless of what you ask for' (roboticape.com sprite-generation post-mortem); png output_format does not change this. Workaround A (chroma key): prompt background as 'EXACT hex #00FF00 (RGB 0, 255, 0) with NO gradients, NO noise, NO texture', 'Add a clean white outline/border 2-3 pixels wide around every sprite', 'No green anywhere in the sprites'; key out with HSV hue within ±22° of 120°, saturation > 0.3, value > 0.3. Workaround B (triangulation matting, jidefr.medium.com): generate on white, second edit call 'Convert the white background to pure black. Preserve all foreground elements.', alpha = 1 - (white_pixel - black_pixel)/255, foreground color recovered 70% from black pass / 30% from white pass; 'costs 2x a normal generation' and 4K 'currently times out' (cap 2K); occasional failure when the model subtly alters the foreground between passes. Community wrappers exist: nano-banana-2-transparent and nano-banana-2-bg-remove.
- Character consistency (reviews): with a single input reference image, Nano Banana 2 'maintained clear character recognition when the same prompt was run about 10 times' (aitoolssme.com 4-prompt head-to-head); supports 'visual consistency for up to 5 characters and detail fidelity for up to 14 objects within a single workflow'; comparison verdict: 'Use Nano Banana 2 for speed, realism, and character consistency', keep Nano Banana Pro for 'more artistic restraint'. Pro accepts the same 14 reference images and is preferred for few-shot style-guide loading (turnaround sheets, palettes).
- Sprite-sheet frame/grid data (roboticape.com): 'Nano Banana does not space sprites in a perfectly uniform grid, even when you ask for one' — slice by detecting 'connected non-transparent regions' instead of fixed-cell division; 'One character per sheet (or two at most)'; recommended density 3-6 items per row, 2-3 rows per sheet max; 'CRITICAL' + capital letters 'gets the model's attention for non-negotiable requirements'; 'Pre-generating colored variants is more files but zero runtime cost and looks much better' (vs runtime tinting).
- Animation-frame consistency (dev.to Gemini pixel-art experiment): simple cycles (idle, walk) work, but 'once you specify complex and continuous actions like attack combos, it suddenly becomes difficult to maintain movement consistency' — fix is to 'break down the flow and give instructions frame by frame. Subdivide the prompts into steps like raise the sword and swing down'; conversational follow-ups preserve identity: 'By continuing the chat, you can successively derive different action modules based on the same character' and 'Keep this part unchanged, but fix only this specific area'; most reliable transparency path there was 'make the background completely black (or a solid color like green)' + manual keying.
- Turnaround-sheet trick for identity lock: generating one image containing three views — 'a direct frontal shot, a 45° profile, and a full 90° side profile' — gives the model 'a complete 3D understanding of the character's head structure' for later consistent generations (Nano Banana character-consistency guides).
- Rosebud.ai sprite workflow prompt template: 'Create a 6-frame side-view walk cycle of a hero character. Horizontal sprite sheet. Transparent background. Consistent proportions.' with refinement prompts 'Align all frames on the same baseline', 'Keep lighting consistent', 'Match exact character proportions' — but note their 'transparent background' request still yields a fake (flat) background per the alpha-channel findings above.

## Actionable for Bloodgem

- Use google/nano-banana-2 at resolution '1K' + output_format 'png' + aspect_ratio '1:1' for Bloodgem sprite work: $0.067/image means a full character animation set (~10-15 sheet generations plus retries) costs roughly $1-2; only bump to 2K ($0.101) for final atlases or large bosses, and avoid 4K (slow, $0.151, times out in transparency pipelines).
- Build the pipeline in two stages: (1) generate ONE canonical character turnaround sheet per character (front / 45-degree / 90-degree side, single image, 'Bloodborne-inspired gothic hunter, top-down 3/4 aerial view game sprite, painterly dark fantasy') and store it; (2) pass that sheet URL in image_input on every subsequent generation ('Using the character in the reference image, generate...'). Reviews show single-reference identity holds across ~10 generations, which is exactly the sprite-sheet regime.
- Do NOT rely on prompting for transparent backgrounds — the model outputs flat RGB even as PNG. For Bloodgem's crisp top-down sprites, adopt the chroma-key recipe: background 'EXACT hex #00FF00 (RGB 0, 255, 0) with NO gradients, NO noise, NO texture', 'clean white outline 2-3 pixels wide around every sprite', 'No green anywhere in the sprites'; key in a Node/canvas post-process with HSV hue 120 +/- 22 deg, sat > 0.3, val > 0.3. Since Bloodgem is Bloodborne-styled (lots of dark reds/browns, little green), green-screen contamination risk is low. Reserve the 2x-cost white/black triangulation-matting pass for assets needing soft alpha (fog, blood mist, particle textures).
- Slice sprite sheets with connected-component detection (after keying, find connected non-transparent regions and their bounding boxes), NOT a fixed grid — the model does not produce uniformly spaced cells even when asked. Normalize each extracted sprite onto a fixed-size canvas anchored at the feet/center for stable top-down registration.
- Keep sheets small and specific: one character per sheet, 3-6 frames per row, max 2-3 rows; describe every frame explicitly ('Frame 1: hunter at rest, cleaver lowered. Frame 2: cleaver raised overhead...'); use 'CRITICAL:' + ALL CAPS for non-negotiables (camera angle, background color, outline). For attack combos and dodge-rolls, generate frame-by-frame or in 2-3 frame chunks — long action sequences drift.
- Restate the camera every single prompt for the 2.5D aerial view (e.g. 'CRITICAL: TOP-DOWN AERIAL VIEW AT 45 DEGREES, character seen from above, consistent scale') — camera drift is a known failure mode; also generate the 8 movement directions as separate requests against the same reference sheet rather than one giant sheet.
- For palette variants (blood-soaked hunter, elemental weapon glows, enemy recolors), pre-generate variants via edit calls with the base sprite in image_input instead of runtime canvas tinting — practitioners report tinting flat RGB sprites looks worse and pre-generation is zero runtime cost.
- Engineering setup: POST to https://api.replicate.com/v1/models/google/nano-banana-2/predictions with 'Prefer: wait' for 1K iterations (fast enough for the 60s window), but use polling or webhooks for 2K/4K; fire batches concurrently (limit well under 600 predictions/min); download every replicate.delivery URL immediately to your asset folder since URLs expire in 1 hour; in Node use `const output = await replicate.run('google/nano-banana-2', { input })` and `writeFile(path, output)` (single FileOutput, not array).
- If a character's canonical sheet needs maximum fidelity (main hunter, key bosses), consider generating that one sheet with google/nano-banana-pro (same 14-image reference input, better few-shot style adherence), then use the cheaper/faster nano-banana-2 for the volume of animation frames referencing it.
